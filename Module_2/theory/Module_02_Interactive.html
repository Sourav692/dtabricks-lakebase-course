2.1 — Separated Compute & Storage

In classic Postgres, compute and storage are one process on one disk. Lakebase splits them. The compute node is stateless past its in-memory cache; durability lives in the storage layer. Press Write Row and follow the WAL record through the system.

Idle. Click Write Row. The WAL record will travel: compute → safekeepers (quorum) → page server → object storage. Notice that compute holds nothing durable — only a soft page cache.

Why this matters. Crash compute. The data survives — because durability moved out of the box that died. That single move is what unlocks branches, scale-to-zero, and zero-lag replicas. Watch for it across the next four panels.

2.2 — Branches as Copy-on-Write

Because the data lives in shared object storage, a new branch is just a new pointer into the existing WAL — not a copy of the bytes. Build a branch tree below. Watch storage cost barely move while the branch count climbs.

The cost insight. Five branches with no diverging writes cost the same as one. That's why every PR can have its own database, every nightly CI run gets a fresh isolated copy, and a "what did the table look like at 09:42" question becomes a single API call.

2.3 — Scale-to-Zero & the Cold-Start Tax

When a Lakebase instance idles past its threshold, compute deprovisions. Storage stays put. The next query wakes compute — boot, attach, replay WAL, then serve. Set the workload below and fire a query.

2.4 — HA & Read Replicas

HA means a sync standby ready to take writes. A read replica is a separate compute pointing at the same storage — no replication lag, no extra storage cost. Toggle the topology, kill an AZ, scale replicas, watch the QPS land.

Don't conflate the two. HA buys you write availability across an AZ failure. Read replicas buy you read throughput. They're orthogonal — production usually wants both.

2.5 — OAuth Token TTL Trap

Lakebase auth uses short-lived OAuth tokens (~1 hour). If your app fetches one at startup and reuses it, you have a working service that goes broken at the 60-minute mark. Press Run and let the clock advance.

Demo · Two-Path Token Lifecycle
Run the clock — watch one path break, one survive
Speed 6×
Idle. Press Run clock. The good path rotates its token every connection via a credential provider. The bad path uses a token cached at app startup — and dies the moment the clock crosses 60 min.

The fix. Wire your driver to a credential provider that mints a fresh token on every connection (or every N minutes, well below TTL). Long-lived pools should validate-on-checkout and rotate. Never hardcode a token. Never cache one past its TTL.
