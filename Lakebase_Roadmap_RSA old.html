▸ Module One · Foundations

What Lakebase actually is

A managed Postgres engine, fused into the Lakehouse. Why Databricks built it, what problem it kills, and how it differs from "yet another database."

Time ~30 minutes Format Theory primer You will learn • Why OLTP matters in the Lakehouse
• Autoscaling vs Provisioned
• Core terminology: project, branch, compute
• The 5 use cases Databricks targets

The one-paragraph pitch

Lakebase is a fully managed, cloud-native PostgreSQL database that lives natively inside the Databricks Data Intelligence Platform. It brings online transaction processing (OLTP) to the Lakehouse — meaning you can finally run the operational, low-latency, high-concurrency, row-level workloads (apps, agents, feature serving, user-facing dashboards) on the same governance plane as your analytical Delta tables, without standing up a separate RDS / Aurora / Cloud SQL stack and without writing reverse-ETL pipelines to bridge the two worlds.

Why Databricks built this

For a decade the Lakehouse has been brilliant at analytical workloads — Delta + Spark + photon. But every real product needs an operational layer too: the user clicks "buy", the agent writes a memory, the app reads the latest profile in 5 ms. Historically customers solved this with a sidecar Postgres, Aurora, or DynamoDB and then stitched the two halves with Fivetran, Debezium, or hand-written CDC. Lakebase removes the seam.

  • Same governance: Lakebase databases register in Unity Catalog as first-class catalogs.
  • Same identity: Workspace users, SCIM groups, OAuth — all native.
  • Same data plane: Synced tables push UC Delta data into Postgres in seconds; Lakehouse Sync (Beta) does the reverse.
  • Same billing: Lakebase compute and storage are Databricks DBUs.

Two flavors: Autoscaling vs Provisioned

As of March 12, 2026, all newly created Lakebase instances are provisioned as Lakebase Autoscaling projects. The older Lakebase Provisioned tier still exists for backward compatibility but is in maintenance mode — new feature work targets Autoscaling.

The five killer use cases

  1. Operational app backends — internal tools, CRUD apps, admin dashboards backed by Databricks Apps.
  2. Real-time feature serving — sub-10ms feature lookups for ML inference.
  3. AI agent state & memory — durable conversation history, tool-call logs, vector-retrievable memories.
  4. Vector search at app latency — pgvector-backed RAG without a third-party vector DB.
  5. Reverse-ETL replacement — synced tables push UC data into Postgres for low-latency reads, eliminating Hightouch / Census pipelines.

Checkpoint

Before moving on, you should be able to answer: "What does Lakebase replace in a typical customer's stack?" — Hint: a managed Postgres + a reverse-ETL tool + (often) a vector DB and an agent state store.

▸ Module Two · Architecture

How it works under the hood

Separated compute and storage, copy-on-write branches, scale-to-zero suspension. The architectural choices that make Lakebase feel different from RDS.

Time ~45 minutes Format Theory + diagrams Key concepts • Compute / storage separation
• Branches as Git for data
• Cold-start mechanics
• HA topology
• Networking & Private Link

The defining architectural move

Lakebase Autoscaling separates compute from storage. Storage is a multi-tenant page server backed by object storage; compute is a Postgres instance that lazily fetches pages from the page server and caches them locally. This decoupling is what enables every magical capability:

  • Scale-to-zero — when no queries run for ~5 minutes, compute suspends. Storage stays. Cost drops to storage-only.
  • Branching — a new branch is just a new compute pointing at a copy-on-write snapshot of storage. No data is copied. Ready in seconds.
  • Instant restore — every page version is retained for the restore window (up to 30 days). Restoring is creating a branch at timestamp T.
  • Read replicas — additional compute nodes share the same storage layer. Zero replication lag for committed pages.
▸ Lakebase Autoscaling Topology

Cold-start mechanics

Scale-to-zero is real but not free. When a suspended branch receives its first connection, Postgres must boot, attach to storage, and warm essential pages. Typical cold-start: ~3–8 seconds for a small DB. The pgbouncer pool keeps the user-facing connection waiting transparently. For latency-critical paths (a real-time recommendation API), disable scale-to-zero on the production branch and let dev/preview branches sleep.

High availability

HA pairs a primary compute in AZ-A with a synchronous standby in AZ-B. Storage is already replicated; HA only protects the compute layer. Failover is typically <30 s and connection strings are stable through it. For RPO=0 critical workloads, enable HA on the primary branch only — branches inherit HA settings independently.

Networking

Three connection paths:

  • Public endpoint over TLS — default; OAuth tokens via Databricks SDK.
  • Inbound Private Link — VPC interface endpoint for VPC-to-VPC traffic without leaving the AWS backbone.
  • Performance-intensive Private Link — dedicated path for high-QPS workloads (Autoscaling only).
Pitfall Lakebase uses short-lived OAuth tokens (typically 1 hour). Long-running app pools must refresh tokens transparently — the Databricks Python / JDBC drivers do this; raw psycopg2 with a hardcoded password will silently fail after an hour. We'll handle this properly in Module 3.

Hands-on · Synced table

Suppose you have a main.features.user_profile Delta table updated every hour by a job, and your app needs to read the latest profile in <10 ms. A synced table replicates Delta into Lakebase using the platform's managed CDC.

synced_table.pyPython
from databricks.sdk.service.database import (
    SyncedDatabaseTable, SyncedTableSpec, SyncedTableSchedulingPolicy
)

st = w.database.create_synced_database_table(
    synced_table=SyncedDatabaseTable(
        name="rsa_lakebase.public.user_profile_synced",
        spec=SyncedTableSpec(
            source_table_full_name="main.features.user_profile",
            primary_key_columns=["user_id"],
            scheduling_policy=SyncedTableSchedulingPolicy.CONTINUOUS,
            timeseries_key="updated_at",
        ),
        database_instance_name="rsa-demo-lakebase",
    )
)

Two scheduling policies are available: CONTINUOUS (streaming, ~seconds latency, higher cost) and SNAPSHOT (scheduled refresh, minutes latency, cheaper). Always start customers on SNAPSHOT and only upgrade to CONTINUOUS when they prove the latency need.

Hands-on · Lakehouse Sync (reverse direction)

For analytics on operational events — "show me the funnel of orders placed in the last hour" — you want Postgres rows in Delta, queryable by Spark. Lakehouse Sync streams logical-replication CDC into Delta tables.

lakehouse_sync.sqlSQL · DBSQL
-- Beta on AWS as of 2026.04 - check region availability
CREATE LAKEHOUSE_SYNC orders_to_delta
FROM DATABASE_INSTANCE 'rsa-demo-lakebase'
SOURCE public.orders
TARGET main.bronze.orders_cdc;

Federation · the lazy option

If volume is low and freshness is critical, just federate. DBSQL queries Lakebase live over the wire. Good for a quarterly auditor's report; bad for a 10M-row join.

RSA Heuristic Federation is "free" but pushes load onto the OLTP. If a customer asks for federation on a hot production DB, propose a synced table or Lakehouse Sync instead — they'll thank you when their app keeps responding under analyst load.

Why use Lakebase for vectors?

Databricks already ships Mosaic AI Vector Search — a purpose-built, scaled vector index that indexes Delta tables. It is the right answer when you have millions to billions of vectors and need the highest recall at scale. So when does Lakebase + pgvector beat it?

  • Transactional consistency — embeddings live in the same Postgres row as the source document, with the same ACID guarantees. No "did the index catch up?"
  • Hybrid SQL filters — combine vector similarity with arbitrary JOINs, ROW LEVEL SECURITY, and rich SQL predicates in a single query.
  • Cost & latency for small / medium corpora — under ~5M vectors, a properly-indexed Postgres usually beats a separate vector service on TCO and end-to-end latency for app queries.
  • Single connection from a Databricks App — your app already needs Postgres for users/sessions; co-locate the embeddings.
Decision rule <5M vectors, app-tier latency, hybrid SQL filters → Lakebase + pgvector.
>10M vectors, dedicated retrieval service, governance via UC → Mosaic AI Vector Search.
Between 5–10M: benchmark both.

Schema design for embeddings

vector_schema.sqlSQL
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE documents (
    id           BIGSERIAL PRIMARY KEY,
    source       TEXT,
    title        TEXT,
    content      TEXT,
    metadata     JSONB,
    embedding    vector(1024),    -- bge-large-en-v1.5 dim
    tsv          tsvector GENERATED ALWAYS AS
                 (to_tsvector('english', coalesce(title,'') || ' ' || coalesce(content,''))) STORED,
    created_at   TIMESTAMPTZ DEFAULT now()
);

-- HNSW for ANN search; m=16, ef_construction=64 are sane defaults
CREATE INDEX idx_doc_embed_hnsw
ON documents USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- Inverted index for keyword search (hybrid retrieval)
CREATE INDEX idx_doc_tsv ON documents USING GIN (tsv);

-- Filter pushdown index
CREATE INDEX idx_doc_meta ON documents USING GIN (metadata);

Generating embeddings with Foundation Models

Use the Databricks Foundation Models API — no separate inference infra. The databricks-bge-large-en model returns 1024-dim vectors and is billed per token.

from databricks.sdk import WorkspaceClient
from sqlalchemy import text

def embed(texts: list[str]) -> list[list[float]]:
    client = w.serving_endpoints.get_open_ai_client()
    resp = client.embeddings.create(
        model="databricks-bge-large-en",
        input=texts,

Querying · pure vector vs hybrid

retrieve.sqlSQL
-- Pure vector top-k
SELECT id, title, content,
       embedding <=> :query_vec AS distance
FROM documents
WHERE metadata->>'lang' = 'en'
ORDER BY embedding <=> :query_vec
LIMIT 10;

-- Hybrid: 70% vector + 30% BM25 (Reciprocal Rank Fusion)
WITH v AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> :qv) AS rk
  FROM documents ORDER BY embedding <=> :qv LIMIT 50
),
k AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(tsv, plainto_tsquery(:q)) DESC) AS rk
  FROM documents WHERE tsv @@ plainto_tsquery(:q) LIMIT 50
)
SELECT d.id, d.title,
       1.0/(60+v.rk) + 1.0/(60+k.rk) AS rrf
FROM documents d
LEFT JOIN v USING(id) LEFT JOIN k USING(id)
WHERE v.id IS NOT NULL OR k.id IS NOT NULL
ORDER BY rrf DESC LIMIT 10;

End-to-end RAG function

Wire it together: a single Python function that takes a question, embeds it, runs hybrid retrieval, and feeds the top-5 chunks into databricks-meta-llama-3-3-70b-instruct. Less than 40 lines.

rag.pyPython
def rag(question: str, k: int = 5) -> str:
    qv = embed([question])[0]
    with engine.connect() as conn:
        rows = conn.execute(text("""
          SELECT title, content
          FROM documents
          ORDER BY embedding <=> :qv
          LIMIT :k
        """), {"qv": str(qv), "k": k}).fetchall()

    context = "\n\n".join(f"# {r.title}\n{r.content}" for r in rows)
    client = w.serving_endpoints.get_open_ai_client()
    out = client.chat.completions.create(
        model="databricks-meta-llama-3-3-70b-instruct",
        messages=[
          {"role":"system","content":"Answer using ONLY the context."},
          {"role":"user",
           "content":f"Context:\n{context}\n\nQ: {question}"}
        ],
        temperature=0.1,
    )
    return out.choices[0].message.content

print(rag("How does Lakebase branching work?"))

The memory contract

Wrap the schema in a clean Python interface so the agent never writes raw SQL:

memory.pyPython
from dataclasses import dataclass
from sqlalchemy import text
import uuid, json

class AgentMemory:
    def __init__(self, engine, embed_fn):
        self.e, self.embed = engine, embed_fn

    def start_session(self, user_id, agent="assistant") -> str:
        with self.e.begin() as c:
            sid = c.execute(text("""
              INSERT INTO sessions(user_id, agent_name)
              VALUES (:u,:a) RETURNING id
            """), dict(u=user_id, a=agent)).scalar()
        return str(sid)

    def add_message(self, sid, role, content, tool_call=None):
        with self.e.begin() as c:
            c.execute(text("""
              INSERT INTO messages(session_id,role,content,tool_call)
              VALUES (:s,:r,:c,:t)
            """), dict(s=sid, r=role, c=content,
                      t=json.dumps(tool_call) if tool_call else None))

    def recent_messages(self, sid, n=20):
        with self.e.connect() as c:
            return c.execute(text("""
              SELECT role, content FROM messages
              WHERE session_id = :s ORDER BY created_at DESC LIMIT :n
            """), dict(s=sid, n=n)).fetchall()[::-1]

    def remember(self, user_id, summary, importance=0.5):
        v = self.embed([summary])[0]
        with self.e.begin() as c:
            c.execute(text("""
              INSERT INTO episodes(user_id, summary, embedding, importance)
              VALUES (:u, :s, :e, :i)
            """), dict(u=user_id, s=summary, e=str(v), i=importance))

    def recall(self, user_id, query, k=5):
        qv = self.embed([query])[0]
        with self.e.connect() as c:
            return c.execute(text("""
              SELECT summary, importance,
                     embedding <=> :q AS dist
              FROM episodes
              WHERE user_id = :u
              ORDER BY (embedding <=> :q) - importance*0.1
              LIMIT :k
            """), dict(u=user_id, q=str(qv), k=k)).fetchall()
Production tip Run a nightly job (Databricks Workflow) that distills long sessions into episodes rows: "User A prefers concise answers, works on Snowflake migrations, runs Postgres 17." The next session starts with high-signal context retrieved by recall().

Wiring into Mosaic AI Agent Framework

If the customer is using Mosaic AI Agent Framework with MLflow, the same Lakebase becomes the persistence layer for the framework's checkpoint store. The state of a multi-step agent (LangGraph nodes, tool plans) survives restarts, scales, and is auditable from any DBSQL editor.

Hands-on challenge

Build a 30-line agent loop that: (1) starts a session, (2) recalls top-3 episodes, (3) runs an LLM turn with that context, (4) writes the assistant's reply, (5) at session end, asks the LLM to summarize the conversation and writes a new episode. You now have a Lakebase-native memory layer.

Why this combination matters

Before Apps + Lakebase, a "data app" on Databricks meant: deploy a container on EKS, point it at an external Postgres, configure SSO separately, manage secrets, and convince security review. Now: databricks apps deploy, the runtime injects Lakebase credentials at startup, identity is the workspace user, every interaction shows up in audit logs.

The runtime contract

When you bind a Lakebase project to an App, the runtime makes three environment variables available to your container:

  • PGHOST — the read-write DNS for the bound project
  • PGUSER — the OAuth-on-behalf-of user (the end user, not a service account)
  • DATABRICKS_TOKEN — short-lived OBO token used as PGPASSWORD

This is the killer detail: the database connection is opened as the end user, not as the app. Postgres GRANTs and RLS policies run against the actual workspace identity. Secure-by-default; no service account to leak.

Patterns we see in the field

  • Internal admin tools — replace Retool / Hightouch UI with Lakebase + Streamlit, governed by UC.
  • AI-powered chat over data — Apps + Lakebase memory + Foundation Models = ChatGPT-for-your-data without third-party SaaS.
  • Approval workflows — orders/expenses sit in Lakebase, an App provides the approval UI, a Job moves rows to main.silver on approval.
  • Customer 360 lookups — synced tables feed Lakebase from UC; an App provides <100ms search across millions of customers.

Connection pooling and the OBO gotcha

Because credentials are scoped per request, the naive pattern (one global engine for the app) breaks: every user gets the first user's identity. Two correct patterns:

  1. Per-request engine — recreate engine per request from the request-scoped token. Simple, slightly slower.
  2. Pool keyed by user — a small LRU of (user, engine) pairs with TTL = token lifetime. Best for high-QPS Apps.
RSA Trap Customers will copy a "global engine" pattern from a tutorial and ship it. You'll find out when their security team finds rows of user A appearing for user B. Always review their connection management on the first PR.

The agent loop · the one piece worth seeing in full

The other phases reuse code patterns from earlier modules. The agent loop is where everything composes — read it carefully:

app/agent.pyPython
import json
from .memory import AgentMemory
from .retrieve import retrieve

TOOLS = [
  {"type":"function","function":{
    "name":"retrieve",
    "description":"Search the support knowledge base.",
    "parameters":{"type":"object",
      "properties":{"query":{"type":"string"}},
      "required":["query"]}}},
  {"type":"function","function":{
    "name":"get_orders",
    "description":"Fetch recent orders for a customer email.",
    "parameters":{"type":"object",
      "properties":{"email":{"type":"string"}},
      "required":["email"]}}},
]

def run_turn(engine, embed_fn, llm, sid, user_id, user_msg):
    mem = AgentMemory(engine, embed_fn)

    # 1. Recall long-term episodes
    episodes = mem.recall(user_id, user_msg, k=3)
    epi_text = "\n".join(f"- {e.summary}" for e in episodes)

    # 2. Build the message stack
    history = mem.recent_messages(sid, n=10)
    messages = [
      {"role":"system",
       "content": f"You are a Northwind support agent assistant. "
                  f"Known about the user:\n{epi_text}"},
      *[{"role":m.role,"content":m.content} for m in history],
      {"role":"user","content":user_msg},
    ]
    mem.add_message(sid, "user", user_msg)

    # 3. Tool-using loop (max 4 iterations)
    for _ in range(4):
        out = llm.chat.completions.create(
            model="databricks-meta-llama-3-3-70b-instruct",
            messages=messages, tools=TOOLS, temperature=0.1)
        msg = out.choices[0].message

        if not msg.tool_calls:
            mem.add_message(sid, "assistant", msg.content)
            return msg.content

        messages.append(msg)
        for tc in msg.tool_calls:
            args = json.loads(tc.function.arguments)
            if tc.function.name == "retrieve":
                result = retrieve(engine, embed_fn, args["query"])
            elif tc.function.name == "get_orders":
                result = _get_orders(engine, args["email"])
            messages.append({"role":"tool",
                             "tool_call_id":tc.id,
                             "content":json.dumps(result)})
            mem.log_tool_call(sid, tc.function.name, args, result)

    return "(reached tool-call limit)"

Best practices · the short list

Schema & query

  • Use BIGSERIAL / BIGINT primary keys, not UUIDs as PKs (UUIDs as secondary keys are fine). Index locality matters more on Lakebase than on a beefy single-node Postgres because pages are remote.
  • Prefer BRIN over B-tree for monotonically-increasing columns (timestamps, sequences). 100x smaller, identical query plans for range scans.
  • Avoid SELECT * from synced tables. The page server fetches whole rows; project the columns you need.
  • Batch writes in transactions of 1k–10k rows. Many tiny commits inflate WAL.

Indexing

  • HNSW for vectors over IVFFlat. Higher build cost, dramatically better recall/latency for app workloads.
  • Partial indexes on hot predicates: WHERE deleted_at IS NULL.
  • JSONB GIN indexes only on columns you query — they're expensive to maintain.

Connections

  • Always use connection pooling. Lakebase's bundled pgbouncer caps at ~1k connections per branch. Naive frameworks (some Node ORMs) open a connection per request — they will exhaust the pool.
  • Refresh OAuth tokens proactively. Tokens are ~1h; refresh at 50 min. The Databricks Python and JDBC drivers do this; raw clients do not.
  • Disable scale-to-zero on prod primary. Cold starts ruin p99 SLAs.

Branching workflow

  • One branch per PR for schema changes. Merge = run migration on main.
  • Ephemeral branches in CI auto-deleted on PR close.
  • Never run perf tests on main. Branch, test, drop.

Cost

  • Tag everything with cost center, env, owner. Use UC tags + Lakebase tags for chargeback.
  • Set retention to the minimum business-acceptable window (default 7d, max 30d). Storage cost is per page-version × time.
  • Aggregate cold tables into Delta via Lakehouse Sync, then drop in Postgres.

When to use Lakebase · decision tree

Strong YES Customer is on Databricks, building an app or agent that needs operational state, vector retrieval, or low-latency feature serving. Wants single bill, single governance, single auth. Vector corpus < 10M. Postgres is acceptable.
Mixed Customer has a 100M-row OLTP on Aurora today. Migration cost is real; Lakebase doesn't make their existing app faster. Recommend Lakebase for new apps in the same workspace — coexist, don't migrate-for-migration's-sake.
Honest NO Customer needs MySQL, SQL Server, or Oracle compatibility. Workload exceeds the published limits (e.g., > few TB hot working set, > 50k QPS, billion-vector index, regions Lakebase doesn't yet support). Don't force-fit; recommend the right tool, retain the trust.

Why Lakebase · the one-sentence pitch you'll give 100 times

"Lakebase is the operational database that finally lives in the same governance plane as your analytical lakehouse — so your apps, agents, and features stop being second-class citizens of your data platform."

Anti-patterns to refuse politely

  • "Use Lakebase as our primary 50TB Aurora replacement next quarter" — too aggressive. Pilot a single new app first.
  • "Run our analytics dashboards directly off Lakebase to skip Delta" — Postgres is OLTP. Push aggregations to Delta + DBSQL; serve hot reads from Lakebase.
  • "Store 1 billion document vectors here" — use Mosaic AI Vector Search. Lakebase + pgvector is for <10M.
  • "Avoid Unity Catalog registration to keep things simple" — without UC, you lose governance, federation, AI/BI access. Always register.
  • "Hardcode the Postgres password in our app config" — tokens expire. Use the SDK's credential provider every time.

The 30-second customer answer

When a customer asks: "Why Lakebase instead of just running Postgres on Aurora?"

  1. Governance. Lakebase tables register in Unity Catalog. Same RBAC, same tags, same audit.
  2. Zero ETL. Synced tables eliminate reverse-ETL pipelines from Delta to your app DB.
  3. Identity. Apps connect on behalf of the end user — RLS works against workspace identity for free.
  4. Branching. Schema changes get tested on copy-on-write branches in seconds, not on a staging RDS that was last refreshed 6 weeks ago.
  5. One bill, one platform. Compute, storage, vectors, models, apps — all on Databricks.

If none of those five matter to the customer, Aurora is the right answer. Tell them so.
