Theory

Module 02
Beg–Int · ~2.5h
Architecture Deep Dive — Compute, Storage, Branches
The architectural choices that make Lakebase feel different from RDS — separated compute and storage, copy-on-write branching, scale-to-zero, HA, and networking.
▶
2.1 Separated Compute & Storage — The Defining Move
Theory
Storage: multi-tenant page server backed by object storage; durable, versioned, copy-on-write
Compute: a Postgres process that lazily fetches pages and caches them locally
Why this matters: every "magical" capability (branches, instant restore, scale-to-zero) flows from this separation
Trade-off: cold pages incur a fetch from the page server — index design matters more than on a single-node Postgres
Mental model: think Neon/Aurora-style architecture, deeply integrated with Databricks identity and UC
2.2 Branches — Git-Like Workflows for Data
TheoryNew
A branch = new compute pointing at a copy-on-write snapshot of storage; ready in seconds, no data copied
Use cases: PR-based schema migrations, ephemeral CI databases, blue-green deploys, "what-if" analysis
Branches inherit from a parent at a point in time; children diverge as they write
Lifecycle: create → diverge → merge-by-promotion or drop; never copied, always linked
Cost model: pay for branch compute + the deltas vs. parent storage
2.3 Scale-to-Zero & Cold-Start Mechanics
TheoryNew
When idle (~5 min), compute suspends; storage stays warm; cost drops to storage-only
Cold start latency: typical ~3–8s for a small DB; pgbouncer holds the client connection during boot
Production rule: disable scale-to-zero on prod primary; enable it on dev/preview branches
Page warming: first queries pull cold pages from the page server; performance ramps over the first minute
Anti-pattern: latency-sensitive APIs running against a scale-to-zero branch — you'll see p99 spikes
2.4 High Availability & Read Replicas
Theory
HA: primary in AZ-A + synchronous standby in AZ-B; failover <30s; connection strings are stable
Storage is already replicated — HA only protects the compute tier
Read replicas: extra compute nodes sharing the same storage; zero replication lag for committed pages
Pattern: primary handles writes + hot reads; replica absorbs analyst / dashboard queries
Provisioned offering uses "readable secondaries"; Autoscaling has true read replicas
2.5 Networking, Auth & the OAuth Token TTL Trap
Theory
Three connection paths: public TLS, Inbound Private Link, and performance-intensive Private Link
Auth: workspace OAuth token used as the Postgres password (PG_PASSWORD)
Token TTL ≈ 1 hour — the trap: hardcoded passwords silently fail after 60 minutes
Mitigation: Databricks Python and JDBC drivers refresh tokens automatically; raw clients must implement refresh
Compliance security profiles (HIPAA / C5 / TISAX) supported on Autoscaling
Token expiry is the #1 cause of mysterious "connection refused" issues in app pools running for hours. Always use the Databricks credential provider, never hardcode.
🎯
Outcome: You can explain Lakebase's architectural decisions and know when each capability (branches, scale-to-zero, replicas, HA) is the right tool. You understand the OAuth lifecycle and can avoid the most common production trap.

Module 04

Sync, Federate & Unify with Unity Catalog
Three integration patterns: synced tables (UC → PG), Lakehouse Sync (PG → Delta), and federation. Pick the right one for the data direction you need.
▶
4.1 The Three Integration Patterns — When To Use Each
Theory
Synced tables — UC Delta → Lakebase, seconds-to-minutes latency, for serving features and profiles to apps
Lakehouse Sync (Beta on AWS) — Lakebase → Delta via continuous CDC, for analytics on operational data
Federation — DBSQL queries Lakebase live; per-query latency, for ad-hoc joins and exploration
Decision rule: write direction + freshness need → synced tables. Read for analytics → Lakehouse Sync. Occasional joins → federation.
Anti-pattern: federation for high-volume reads — pushes load onto your OLTP and degrades app latency
4.2 Lab: Create a Synced Table from a Delta Source
Hands-OnNew

Replicate a UC Delta table into Lakebase so an app can read at app latency.

Pre-req: a Delta table in main.features.user_profile with primary key user_id
Create with w.database.create_synced_database_table()
Choose scheduling: SNAPSHOT (cheap, minutes) vs. CONTINUOUS (streaming, seconds)
Verify: insert a row in Delta → SELECT it from Lakebase within seconds
Add Postgres-side indexes after sync — the synced table is a read mirror, but you query it like any table
from databricks.sdk.service.database import ( SyncedDatabaseTable, SyncedTableSpec, SyncedTableSchedulingPolicy ) st = w.database.create_synced_database_table( synced_table=SyncedDatabaseTable( name="lakebase_tutorial.public.user_profile_synced", spec=SyncedTableSpec( source_table_full_name="main.features.user_profile", primary_key_columns=["user_id"], scheduling_policy=SyncedTableSchedulingPolicy.CONTINUOUS, timeseries_key="updated_at", ), database_instance_name="lakebase-tutorial", ) )
4.3 Lab: Stream Lakebase Changes Back to Delta (Lakehouse Sync)
Hands-OnNew

Capture operational events into Delta for analytics — without rolling your own Debezium.

Beta on AWS as of 2026.04 — verify region availability before designing for prod
Create with SQL: CREATE LAKEHOUSE_SYNC ... FROM DATABASE_INSTANCE ... SOURCE ... TARGET ...
Underlying mechanism: Postgres logical replication slots → managed CDC → Delta
Inspect target: a bronze Delta table arrives with all CDC columns (_change_type, _commit_version)
Build a silver layer with Delta MERGE for the latest state per primary key
-- Run in DBSQL CREATE LAKEHOUSE_SYNC orders_to_delta FROM DATABASE_INSTANCE 'lakebase-tutorial' SOURCE public.orders TARGET main.bronze.orders_cdc; -- Build current-state silver via MERGE on _change_type MERGE INTO main.silver.orders s USING ( SELECT * FROM main.bronze.orders_cdc QUALIFY row_number() OVER (PARTITION BY id ORDER BY _commit_version DESC) = 1 ) c ON s.id = c.id WHEN MATCHED AND c._change_type = 'delete' THEN DELETE WHEN MATCHED THEN UPDATE SET * WHEN NOT MATCHED AND c._change_type <> 'delete' THEN INSERT *;
4.4 Lab: Federated Query — When Live Reads Are Right
Hands-On
After UC registration, Lakebase tables are immediately federable from DBSQL
Join Postgres rows directly with Delta tables: SELECT ... FROM lakebase_tutorial.public.orders o JOIN main.gold.products p ...
Push-down: simple predicates and projections push to Postgres; complex aggregations may pull rows
Use case: quarterly auditor's one-off report; bad fit for hourly scheduled jobs
RSA heuristic: if you'd run it more than once a week, build a synced table or Lakehouse Sync instead
4.5 Cost & Latency Comparison Between The Three
Theory
Synced tables: pay for the sync compute + Lakebase storage; CONTINUOUS > SNAPSHOT cost-wise
Lakehouse Sync: pay for the CDC pipeline + Delta storage; cheap once steady-state
Federation: zero ongoing cost, but every query taxes the Postgres primary
Latency: synced tables ~seconds; Lakehouse Sync ~tens of seconds; federation = network round-trip
Always start with SNAPSHOT scheduling and only upgrade to CONTINUOUS when latency is proven necessary

Lakebase as a Vector Store — pgvector at App Latency
Production RAG without bolting on a third-party vector DB. When pgvector is right, when Mosaic AI Vector Search is right, and how to build hybrid retrieval correctly.
▶
5.1 When To Use Lakebase + pgvector vs. Mosaic AI Vector Search
Theory
Lakebase + pgvector wins for: <5M vectors, hybrid SQL filters, transactional consistency, single-DB apps
Mosaic AI Vector Search wins for: >10M vectors, dedicated retrieval service, governance via UC, large-scale recall
Between 5M and 10M: benchmark both — query patterns matter more than corpus size
Hybrid filter advantage: pgvector lets you mix vector similarity with arbitrary SQL JOINs and RLS
Co-location advantage: if your app already needs Lakebase for users/sessions, embeddings live one query away
5.2 Schema Design for Embeddings — Indexes That Matter
TheoryNew
Use HNSW over IVFFlat — higher build cost, dramatically better recall/latency for app workloads
HNSW parameters: m=16, ef_construction=64 are sane defaults; tune ef_search per query
Generated tsvector column + GIN index for keyword/BM25 component of hybrid retrieval
JSONB metadata + GIN index for filter pushdown (lang, source, tenant, etc.)
Choose embedding dimension based on model: databricks-bge-large-en = 1024-dim
5.3 Lab: Build the pgvector Schema from Scratch
Hands-On
Create the vector extension: CREATE EXTENSION IF NOT EXISTS vector
Define a documents table with content, JSONB metadata, vector embedding, and generated tsvector
Add HNSW index using cosine distance (vector_cosine_ops)
Add GIN index on tsvector and JSONB metadata
CREATE EXTENSION IF NOT EXISTS vector; CREATE TABLE documents ( id BIGSERIAL PRIMARY KEY, source TEXT, title TEXT, content TEXT, metadata JSONB, embedding vector(1024), tsv tsvector GENERATED ALWAYS AS (to_tsvector('english', coalesce(title,'') || ' ' || coalesce(content,''))) STORED, created_at TIMESTAMPTZ DEFAULT now() ); CREATE INDEX idx_doc_hnsw ON documents USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64); CREATE INDEX idx_doc_tsv ON documents USING GIN (tsv); CREATE INDEX idx_doc_meta ON documents USING GIN (metadata);
5.4 Lab: Embed & Bulk-Load Documents via Foundation Models API
Hands-OnNew

No separate inference infra — Foundation Models gives you embeddings on the same platform.

Use the OpenAI-compatible client from the SDK: w.serving_endpoints.get_open_ai_client()
Embed in batches of 32–64 to balance throughput and rate limits
Insert in transactions of 500–1000 rows to amortize commit cost
For datasets > 1M rows: use COPY via psycopg.copy for 10–50× faster ingest

5.5 Lab: Hybrid Retrieval with Reciprocal Rank Fusion
Hands-OnNew

Vector alone misses keyword matches; BM25 alone misses semantic matches. RRF fuses both.

Run two CTEs: top-50 by vector distance, top-50 by ts_rank
Combine with RRF: 1/(60+rank_v) + 1/(60+rank_k)
Apply metadata filters in the WHERE clause for tenant isolation
Tune ef_search per query for the recall/latency trade-off
WITH v AS ( SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> :qv) AS rk FROM documents ORDER BY embedding <=> :qv LIMIT 50 ), k AS ( SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(tsv, plainto_tsquery(:q)) DESC) AS rk FROM documents WHERE tsv @@ plainto_tsquery(:q) LIMIT 50 ) SELECT d.id, d.title, 1.0/(60+v.rk) + 1.0/(60+k.rk) AS rrf FROM documents d LEFT JOIN v USING(id) LEFT JOIN k USING(id) WHERE v.id IS NOT NULL OR k.id IS NOT NULL ORDER BY rrf DESC LIMIT 10;
5.6 Lab: End-to-End RAG Function in <40 Lines
Hands-On
Embed the question, run the hybrid retrieval, build a context block
Call databricks-meta-llama-3-3-70b-instruct with system prompt + context + question
Set temperature low (0.0–0.2) for grounded answers; raise for creative tasks
Wire it into a notebook UI cell to test interactively
def rag(question: str, k: int = 5) -> str: qv = embed([question])[0] with engine.connect() as c: rows = c.execute(text(""" SELECT title, content FROM documents ORDER BY embedding <=> :qv LIMIT :k """), {"qv": str(qv), "k": k}).fetchall() ctx = "\n\n".join(f"# {r.title}\n{r.content}" for r in rows) out = client.chat.completions.create( model="databricks-meta-llama-3-3-70b-instruct", messages=[ {"role":"system","content":"Answer using ONLY the context."}, {"role":"user", "content":f"Context:\n{ctx}\n\nQ: {question}"}], temperature=0.1) return out.choices[0].message.content

Lakebase as Memory Store for AI Agents
An agent without memory is a chatbot with amnesia. The four memory layers — short-term, episodic, semantic, procedural — and how to model them all in a single Postgres.
▶
6.1 The Four Memory Layers — Conceptual Model
TheoryNew
Short-term — current conversation messages, keyed by session_id
Long-term episodic — distilled memories from past sessions for one user, vector-retrievable
Semantic — cross-user knowledge base (the documents table from Module 5)
Procedural — append-only audit trail of tool calls (args, result, duration, success)
All four can live in the same Lakebase database — one auth, one bill, one query plane
6.2 Lab: Schema for Sessions, Messages, Episodes, Tool Calls
Hands-On
Use UUID primary keys for sessions (clients can generate them client-side)
Foreign key from messages and tool_calls to sessions with ON DELETE CASCADE
Episodes get a vector embedding for retrieval and an importance score for ranking
Add a (session_id, created_at) index on messages for fast history retrieval
CREATE TABLE sessions ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id TEXT NOT NULL, agent_name TEXT, created_at TIMESTAMPTZ DEFAULT now(), ended_at TIMESTAMPTZ ); CREATE TABLE messages ( id BIGSERIAL PRIMARY KEY, session_id UUID REFERENCES sessions(id) ON DELETE CASCADE, role TEXT CHECK (role IN ('system','user','assistant','tool')), content TEXT, tool_call JSONB, created_at TIMESTAMPTZ DEFAULT now() ); CREATE INDEX ON messages(session_id, created_at); CREATE TABLE episodes ( id BIGSERIAL PRIMARY KEY, user_id TEXT NOT NULL, summary TEXT, embedding vector(1024), importance REAL DEFAULT 0.5, last_seen TIMESTAMPTZ DEFAULT now() ); CREATE INDEX ON episodes USING hnsw (embedding vector_cosine_ops); CREATE INDEX ON episodes(user_id); CREATE TABLE tool_calls ( id BIGSERIAL PRIMARY KEY, session_id UUID REFERENCES sessions(id), tool_name TEXT, args JSONB, result JSONB, duration_ms INT, ok BOOL, ts TIMESTAMPTZ DEFAULT now() );
6.3 Lab: The AgentMemory Class — Clean Python Contract
Hands-OnNew

Wrap raw SQL behind methods so the agent never sees Postgres directly.

Methods: start_session, add_message, recent_messages, remember, recall, log_tool_call
recall uses vector distance + importance bonus for memory ranking
Always parameterized SQL — no string concatenation, no SQL injection risk
Make the class injectable with the engine and embed function for testability
class AgentMemory: def __init__(self, engine, embed_fn): self.e, self.embed = engine, embed_fn def start_session(self, user_id, agent="assistant"): with self.e.begin() as c: sid = c.execute(text(""" INSERT INTO sessions(user_id, agent_name) VALUES (:u,:a) RETURNING id"""), dict(u=user_id, a=agent)).scalar() return str(sid) def remember(self, user_id, summary, importance=0.5): v = self.embed([summary])[0] with self.e.begin() as c: c.execute(text(""" INSERT INTO episodes(user_id, summary, embedding, importance) VALUES (:u, :s, :e, :i)"""), dict(u=user_id, s=summary, e=str(v), i=importance)) def recall(self, user_id, query, k=5): qv = self.embed([query])[0] with self.e.connect() as c: return c.execute(text(""" SELECT summary, importance, embedding <=> :q AS dist FROM episodes WHERE user_id = :u ORDER BY (embedding <=> :q) - importance*0.1 LIMIT :k"""), dict(u=user_id, q=str(qv), k=k)).fetchall()

7.1 Why Apps + Lakebase Is Different from "App + Some Postgres"
Theory
Before: deploy a container on EKS, point at external Postgres, configure SSO, manage secrets
After: databricks apps deploy; runtime injects creds; identity is the workspace user
Single bill, single audit, single governance plane — apps are first-class data products

7.2 The OBO Runtime Contract — On-Behalf-Of Auth
TheoryNew
Bound apps receive PGHOST, PGUSER, DATABRICKS_TOKEN, PGDATABASE as env vars
The token is scoped to the end user, not the app — every connection runs under their identity
Postgres GRANTs and RLS policies operate against the workspace user — secure-by-default
Implication: there is no service account to leak, no shared secret to rotate
The trap: a global SQLAlchemy engine reuses the first user's token for everyone — never do this
The connection-pooling trap is the #1 security incident vector in real-world Apps deployments. Read 7.4 carefully.

7.4 Lab: Per-Request Engine Factory — Doing OBO Correctly
Hands-OnNew

Two correct patterns; pick based on your traffic profile.

Pattern 1 (simple): rebuild the engine per request from the request-scoped token
Pattern 2 (high QPS): keep an LRU cache keyed by (user_id, token_hash), TTL = token lifetime
Always set pool_pre_ping=True to detect token expiry mid-flight
Never share an engine across users — rows from user A will appear for user B

RSA Toolkit — Comparison, Best Practices & Limitations
The conversations customers actually have. Lakebase vs. RDS / Aurora / Neon / Supabase. Limits to flag in design reviews. The decision tree, the anti-patterns, and the one-sentence pitch you'll give 100 times.
▶
9.1 Lakebase vs. The PostgreSQL Competitive Set
Theory
vs. AWS Aurora PG: Aurora has Aurora Serverless v2 + cloning, but no UC integration, no OBO, no Databricks Apps. Aurora wins on raw OLTP scale.
vs. Neon: Both have branching and scale-to-zero. Neon wins on pure-Postgres dev workflow; Lakebase wins on lakehouse integration and governance.
vs. Supabase: Supabase has a richer mobile/JAMstack story (Auth, Edge Functions, Realtime). Lakebase wins for analytical/AI-heavy workloads on Databricks.
vs. RDS / Cloud SQL: Legacy lift-and-shift. Manual scaling, no branching, no UC. Choose for migration simplicity, not capability.
Honest framing: Lakebase is a different product class — a Postgres governed and synchronized as part of a lakehouse. If those properties don't matter to the customer, recommend the alternative without hesitation.
9.2 Best Practices — Schema, Indexing, Connections
Theory
Use BIGSERIAL / BIGINT primary keys, not UUIDs. Index locality matters more on remote-paged storage.
Prefer BRIN over B-tree for monotonically-increasing columns (timestamps, sequences). 100× smaller, equivalent range plans.
Avoid SELECT * from synced tables — page server fetches whole rows; project columns you need.
Batch writes in transactions of 1k–10k rows; tiny commits inflate WAL.
HNSW over IVFFlat for vectors; partial indexes on hot predicates; GIN only where you query.
Always use connection pooling — Lakebase pgbouncer caps ~1k connections per branch.
Refresh OAuth tokens proactively at 50 min; the Databricks driver does this; raw clients do not.
Disable scale-to-zero on prod primary — cold starts ruin p99 SLAs.
9.3 Best Practices — Branching, Cost, Governance
Theory
One branch per PR for schema changes. Merge = run migration on main.
Ephemeral branches in CI auto-deleted on PR close.
Never run perf tests on main. Branch, test, drop.
Tag everything with cost center, env, owner — UC tags + Lakebase tags drive chargeback.
Set retention to the minimum business-acceptable window (default 7d). Storage cost is per page-version × time.
Aggregate cold tables into Delta via Lakehouse Sync, then drop in Postgres.
Always register in UC — without UC, you lose governance, federation, AI/BI access.
9.4 Limitations to Flag in Every Design Review
TheoryNew

The list customers will hit and blame you for not surfacing earlier.

Postgres extensions are a curated allowlist — no pg_cron, no custom builds; use Databricks Workflows for scheduling
No superuser — some ORM defaults fail (e.g., Hibernate auto-DDL); create roles with required GRANTs
Logical replication slots: limited — external CDC tools need careful slot accounting; use Lakehouse Sync where possible
OAuth token TTL ≈ 1 hour — long batch jobs / app pools need refresh logic; chunk batches under 1h
Region availability is not universal — verify before the design call; offer Provisioned where Autoscaling isn't yet GA
Cold starts on scale-to-zero (~3–8s) — disable on prod primary; enable on dev/preview only
Synced table latency seconds–minutes — not "instant" UC → Lakebase consistency; for write-once-read-fast use direct writes
Lakehouse Sync (reverse) is Beta on AWS — for mission-critical CDC, validate region GA status; have fallback
Not a billion-row vector store — switch to Mosaic AI Vector Search above ~10M vectors
No cross-region active-active — global apps need additional architecture; HA across AZs only
9.5 When To Use Lakebase — The Decision Tree
Theory
Strong YES: Customer is on Databricks, building app or agent needing operational state / vector retrieval / low-latency feature serving. Wants single bill, single governance, single auth. Vector corpus < 10M. Postgres acceptable.
Mixed: Customer has 100M-row OLTP on Aurora today. Migration cost is real; Lakebase doesn't make existing app faster. Recommend Lakebase for new apps in same workspace — coexist, don't migrate-for-migration's-sake.
Honest NO: Customer needs MySQL / SQL Server / Oracle compatibility. Workload exceeds limits (TB hot working set, 50k+ QPS, billion-vector index, unsupported region). Don't force-fit; recommend the right tool, retain the trust.
Run every Lakebase customer conversation through this filter before pitching
9.6 The 30-Second Customer Answer
Theory

When a customer asks: "Why Lakebase instead of Aurora?" — five reasons, in order:

Governance. Lakebase tables register in Unity Catalog. Same RBAC, same tags, same audit.
Zero ETL. Synced tables eliminate reverse-ETL pipelines from Delta to your app DB.
Identity. Apps connect on behalf of the end user — RLS works against workspace identity for free.
Branching. Schema changes get tested on copy-on-write branches in seconds, not staging RDS that was last refreshed 6 weeks ago.
One bill, one platform. Compute, storage, vectors, models, apps — all on Databricks.
If none of those five matter to the customer, Aurora is the right answer. Tell them so.
9.7 Anti-Patterns to Refuse Politely
Theory
❌ "Use Lakebase as our 50TB Aurora replacement next quarter" — too aggressive. Pilot a single new app first.
❌ "Run analytics dashboards directly off Lakebase to skip Delta" — Postgres is OLTP. Push aggregations to Delta + DBSQL; serve hot reads from Lakebase.
❌ "Store 1 billion document vectors here" — use Mosaic AI Vector Search. Lakebase + pgvector is for <10M.
❌ "Avoid UC registration to keep things simple" — without UC, you lose governance, federation, AI/BI access. Always register.
❌ "Hardcode the Postgres password in our app config" — tokens expire. Use the SDK's credential provider every time.
❌ "One global SQLAlchemy engine for the whole App" — breaks OBO; rows of user A appear for user B. Per-request engine factory.
